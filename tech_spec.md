# Техническое ТЗ: AI-ассистент психолога (v5)

## 1. Продуктовое позиционирование

**Не** «AI-психолог». **Да** — цифровой ассистент конкретного специалиста для **базы знаний и сопровождения клиентов между сессиями**.

Две ключевые функции продукта:

1. **База знаний по методу психолога** — справочник, доступный психологу и клиентам. Отвечает на вопросы про метод в её стиле, опираясь на её лекции.
2. **Сопровождение между сессиями** — структурированные режимы взаимодействия с клиентом (дневник, подготовка к сессии, рефлексия, домашки), плюс открытое взаимодействие с памятью клиента (в полной версии).

**Что мы НЕ делаем:**
- Первичный приём, скрининг, диагностика — это зона профессионалов.
- Замена терапевта — бот не ведёт самостоятельную терапию.
- Работа с острыми клиентами — только стабильные, давно в терапии.

Психолог — супервизор и ответственное лицо. Бот — инструмент.

## 1a. Стратегия MVP: от узкого ядра к полному видению

Полное ТЗ ниже описывает **финальное видение** продукта. Чтобы дойти до него — не делаем всё сразу. Разбиваем на этапы, каждый из которых даёт **самостоятельную ценность** и снижает риски.

**Фокус продукта:** база знаний по методу психолога + сопровождение клиентов между сессиями. Первичный приём, скрининг, диагностика — **не наша зона**.

### MVP-0: База знаний для психолога (4 недели)

**Самый узкий и безопасный этап. Без клиентов, без сессий.**

Что делаем:
- Обработка лекций психолога через WhisperX → структурированный корпус
- Voice-document v1 (её принципы, red lines, стиль) — на основе лекций и интервью
- Минимальный справочный интерфейс: «как я объясняла Х в лекциях», «какие техники я применяю при Y»
- Доступ только у психолога и разработчика

Что **не делаем**:
- Не обрабатываем сессии
- Не строим граф клиента
- Не делаем бот для клиентов

Самостоятельная ценность: психолог получает поисковый инструмент по своим лекциям, говорящий в её же стиле.

**🚦 Gate перед MVP-1:** psychotherapist говорит «справочник работает, отвечает в моём стиле».

### MVP-1: Case-study на записях разработчика-клиента (8 недель)

**Глубина вместо ширины. Валидация онтологии графа на одном клиенте.**

> Подробности — в разделе 1b. Используются только «закрытые» сессии, разработчик-клиент не пользуется графом для собственной терапии, психолог имеет полное право вето.

Что делаем:
- Транскрипция отобранных закрытых сессий разработчика-клиента
- Совместная ручная разметка с психологом → golden standard
- Автоматический пайплайн извлечения узлов и связей, проверяемый на golden standard
- Стилевая библиотека (обезличенно, на основе её ходов в этих сессиях)
- Supervision UI: визуализация графа, валидация узлов, право вето
- Pre-session report по запросу — психолог использует при подготовке к сессиям с разработчиком-клиентом

Что **не делаем**:
- Не строим бот для клиентов
- Не работаем с активными темами разработчика-клиента
- Не обрабатываем других клиентов

Самостоятельная ценность: валидированная онтология графа + рабочий протокол подключения новых клиентов в будущем + полезный инструмент для подготовки к сессиям с разработчиком-клиентом.

**🚦 Gate перед MVP-2:** онтология валидирована (≥80% совпадение автомата с golden standard), двойная роль не нарушила терапию.

### MVP-2: Минимальное сопровождение между сессиями (6–8 недель)

**Первая работающая система, через которую клиент взаимодействует с ботом. Намеренно ограниченный набор функций — низкий риск, но реальная польза.**

Кто пилотные пользователи:
- **Разработчик-клиент** в первую очередь (сама строит — понимает риски)
- 1–2 действующих клиента психолога, давно в терапии, не-острых, с явным согласием — добавляются по готовности

Что входит:
- Telegram-бот в режимах:
  - **Дневник** (короткая структурированная форма: что чувствую, что произошло, интенсивность, тело)
  - **Подготовка к сессии** (за 24 часа: что хочу обсудить)
  - **Рефлексия после сессии** (через 2–3 часа: что унесла)
  - **Только слушай** (выговориться без ответа)
  - **Кнопка «передай психологу»** — отметить тему для следующей сессии
- Напоминания о домашках, назначенных психологом
- Чат-режим только в формате «вопрос про метод» (использует базу знаний MVP-0) — без интерпретаций, без эмоциональной работы
- Safety Layer v1: regex + Claude Haiku, hardcoded кризисные ответы, уведомление психолога
- Локальные ресурсы экстренной помощи (РФ, Самуи)
- Pre-session report для психолога автоматически перед каждой сессией
- Supervision UI: просмотр диалогов, пометки good/bad, voice-doc editor с версиями
- Лимиты взаимодействия (длительность, частота)
- Базовый мониторинг привязанности

Что **не входит** (отложено в MVP-3):
- Голосовые сообщения от клиента (только текст)
- Анализ эмоций / просодия
- Открытый терапевтический чат (бот не отвечает на «мне плохо, поговори со мной»)
- GraphRAG в живом диалоге (граф используется только для отчётов психологу, не для ответов клиенту)
- Стандартизированные шкалы
- Эмоциональный классификатор

Самостоятельная ценность: клиент получает инструмент для рефлексии и связи с терапией между сессиями, психолог — структурированную картину между встречами. Без открытого терапевтического чата, который несёт основные риски.

**🚦 Gate перед MVP-3:**
- Разработчик-клиент пользуется системой 4+ недель, никаких искажений терапии
- Если подключён действующий клиент психолога — стабильная работа 4+ недель
- 90%+ ответов бота помечены психологом как «good»
- Safety layer ни разу не пропустил кризисный маркер на тестах
- Психолог говорит «готова к расширению»

### MVP-3: Полное сопровождение между сессиями (10–12 недель)

**Расширение MVP-2 до полного видения: голос, GraphRAG в живом диалоге, шкалы.**

Что добавляется:
- Голосовые сообщения от клиента
- Whisper-транскрипция входящих голосовых
- Просодия (темп, паузы, энергия) и baseline на клиента
- Эмоциональный классификатор (опционально, после оценки качества на лекциях)
- GraphRAG в живом диалоге: бот отвечает с учётом графа клиента (а не только базы знаний)
- Стандартизированные шкалы (PHQ-9, GAD-7) раз в 1–2 недели
- Daily check-ins (короткие 1-минутные)
- Открытый чат-режим с полным контекстом — самое рискованное, последним
- Расширенный режим тренажёра (психолог играет «существующего клиента» с историей)
- Anti-sycophancy post-processor на размеченных примерах из реальных диалогов

**🚦 Gate перед каждым новым клиентом в MVP-3:**
- Психолог прошла полный тренажёр на этой версии
- Клиент дал явное согласие, понимает границы
- Психолог готова реагировать на алерты в течение часа

### MVP-4: Доводка и масштаб

После того как MVP-3 стабильно работает 2–3 месяца:
- Мониторинг привязанности на real-data
- Weekly system reports от бота психологу
- A/B voice-document версий
- Расширение клиентской базы
- Возможный второй психолог-партнёр

### Критичная логика последовательности

1. **MVP-0** даёт базу знаний без риска — нет клиентов вообще.
2. **MVP-1** валидирует онтологию графа на одном клиенте — глубина без масштабирования.
3. **MVP-2** — первое реальное взаимодействие клиент↔бот, но в **намеренно безопасных режимах**: дневник, подготовка к сессии, рефлексия. Никакого открытого терапевтического чата.
4. **MVP-3** — открытый чат, голос, граф в живом диалоге. Только когда инфраструктура отработана.

**Главный принцип:** каждый MVP даёт самостоятельную ценность. Если проект остановится на любом этапе — то, что сделано, всё равно полезно.

## 1b. Текущее состояние данных и стартовая корректировка

### Что есть на момент старта

- **Лекции психолога:** большой корпус, готов к обработке.
- **Записи сессий:** только с одним клиентом — **самим разработчиком проекта** (далее «разработчик-клиент»). Терапия с этим клиентом продолжается, новые записи накапливаются.

### Что это меняет в плане

Исходный MVP-1 предполагал данные с разных клиентов для:
- извлечения стилевой библиотеки в разных контекстах
- валидации онтологии графа на разнообразии случаев
- проверки изоляции между клиентами

С одним клиентом этого сделать нельзя. Поэтому MVP-1 **разбивается на два под-этапа**:

**MVP-0:** база знаний из лекций + voice-document. Не требует записей сессий, опирается на лекции и прямые интервью с психологом. Самостоятельная ценность: справочный инструмент по методу психолога.

**MVP-1:** глубокий case-study на записях разработчика-клиента. Цель — **выработать и валидировать онтологию графа** на одном глубоком случае, прежде чем масштабировать. Совместно с психологом: ручная разметка записей → золотой стандарт → автоматический пайплайн извлечения, сверяемый с золотым стандартом.

Это превращает ограничение («один клиент») в преимущество: **глубина вместо ширины**, валидированная онтология, а не сырая.

### Двойная роль: разработчик одновременно клиент

Это структурный риск, который проект обязан адресовать явно.

**Возможные искажения:**

1. **Терапевтический процесс может пострадать.** Слушая записи как «материал», разработчик-клиент перестаёт прорабатывать содержание сессий и начинает их анализировать.
2. **Поведение в сессиях может неосознанно подстраиваться** под то, что хорошо ложится в граф (более чёткие формулировки, структурированные инсайты).
3. **Психолог тоже знает о проекте**, что может изменить её работу в сторону «play to camera».
4. **Разработчик видит свой граф клиента** — это меняет отношение к собственным паттернам вне терапии.
5. **Активные темы становятся данными** — то, что сейчас прорабатывается, не должно одновременно быть учебным материалом.

### Правила работы с записями разработчика-клиента

Эти ограничения встроены в проект **архитектурно**, не как пожелание:

**1. Используются только «закрытые» сессии.**
- Сессия попадает в обработку **не раньше чем через N месяцев** после записи (конкретный срок определяется психологом).
- Дополнительный критерий: психолог явно подтверждает, что тема, обсуждавшаяся в этой сессии, отработана.
- Все остальные записи складываются в архив без обработки.
- Это правило проверяется в пайплайне: при импорте записи требуется метка `closed_by_therapist=true` и `recording_date <= today - N months`.

**2. Граф разработчика-клиента — двойного назначения.**
- Граф используется как инженерный артефакт для разработки и валидации системы.
- Граф также используется ботом в живых диалогах с разработчиком-клиентом (когда она пользуется системой как пилотный пользователь в MVP-2/MVP-3).
- **Разработчик-клиент сознательно принимает это решение**, понимая, что видит свой собственный граф изнутри проекта.
- Главное условие: содержание этого графа никогда не используется для других клиентов и не упоминается в общих материалах проекта.

**2a. Разработчик-клиент — первый пилотный пользователь.**
- В MVP-2 разработчик-клиент использует систему первой, до подключения других клиентов.
- Это сознательный выбор: разработчик понимает риски лучше любого другого пилотного пользователя.
- Регулярная сверка терапевтического состояния (см. правило 4) особенно критична на этом этапе.
- Если в любой момент терапевтический процесс начнёт страдать — разработчик-клиент выходит из роли пилотного пользователя, и пилот переходит на действующих клиентов психолога.

**3. Психолог сохраняет полное право вето.**
- Любая сессия может быть исключена из обработки задним числом, даже если уже размечена.
- Любой узел графа может быть удалён по запросу психолога без обсуждения.
- Если психолог в какой-то момент скажет «эта запись больше не должна быть в системе» — она удаляется полностью, включая все производные узлы и эмбеддинги.

**4. Регулярная сверка состояния (ежемесячно).**
- Разработчик и психолог обсуждают: не пострадала ли терапия, не появились ли искажения, всё ли в порядке с двойной ролью.
- Это встреча отдельная от обычных сессий и от рабочих встреч по проекту.

**5. Граница между «терапия» и «разработка» соблюдается во времени.**
- Во время сессии разработчик-клиент находится в роли клиента. Точка.
- Обработка записей и работа с проектом — в отдельное время, желательно не в день сессии.

**6. Внешний клинический взгляд.**
- Если у психолога есть супервизор — конфигурация проекта обсуждается с ним.
- Если нет — психолог сама привлекает коллегу для регулярного внешнего взгляда на ситуацию (раз в 2–3 месяца).

### Что меняется в архитектуре БД

Добавляется отметка статуса записи и дата готовности к обработке:

```sql
ALTER TABLE raw_transcripts ADD COLUMN therapy_status TEXT 
  DEFAULT 'active';  -- 'active' | 'closed' | 'excluded'
ALTER TABLE raw_transcripts ADD COLUMN eligible_for_processing_at TIMESTAMP;
ALTER TABLE raw_transcripts ADD COLUMN closed_by_therapist_at TIMESTAMP;

-- Пайплайн обработки фильтрует:
--   WHERE therapy_status = 'closed'
--     AND eligible_for_processing_at <= NOW()
--     AND closed_by_therapist_at IS NOT NULL
```

### Когда заканчивается режим «один клиент = разработчик»

Как только в проекте появляется второй клиент с записями (через MVP-2 — pre-screening новых клиентов, или через явное согласие действующего клиента психолога):

- Онтология графа проверяется на новом случае
- Стилевая библиотека пересобирается с учётом разнообразия
- Изоляция между клиентами тестируется впервые на реальных данных

До этого момента все ограничения раздела 1b остаются в силе.

## 3. Стек

- **Python 3.11+ / FastAPI**
- **aiogram** — Telegram
- **PostgreSQL 16 + pgvector** — RAG, граф, логи, голосовые фичи
- **Redis** — очереди, кэш, rate-limiting
- **Claude API** (Sonnet/Opus) — основной инференс
- **Claude Haiku** — safety-классификатор, пост-процессор
- **WhisperX** (Whisper large-v3 + pyannote) — транскрипция + диаризация
- **librosa + parselmouth** — просодические фичи (темп, pitch, паузы, jitter)
- **SpeechBrain / wav2vec2-emotion** — эмоциональная классификация (локально)
- **Celery / RQ** — фоновые задачи
- **Hetzner VPS + Coolify** — деплой, **отдельный инстанс** от Cook It Easy. Coolify импортирует наш `docker-compose.yml`, даёт UI для логов/метрик/бэкапов и one-click сервисы для Postgres+pgvector и Redis
- **GPU-инстанс** (разово или по расписанию) — Whisper + эмоции
- **Sentry** — мониторинг

## 4. Пайплайн подготовки базы знаний

### 4.1 Транскрипция и диаризация

```
audio → WhisperX → {
  raw.json         (слова + таймкоды + speakers + VAD)
  clean.txt        (очищенный текст через LLM)
  metadata.json
}
```

Два уровня вывода: `raw` (с «эээ» и паузами для анализа невербалики) и `clean` (для RAG).

### 4.2 Сегментация

По смыслу, не по длине:
- Лекции → смысловые блоки + заголовок + резюме.
- Сессии → терапевтические эпизоды.

### 4.3 Обогащение (три параллельных прохода)

**А — лекции → понятия метода** (термины, техники, примеры, утверждения).

**Б — сессии → стилевая библиотека (обезличенная):** её вопросы, интервенции, формулировки, ходы. Содержание клиентов не сохраняется.

**В — сессии → граф клиента (персональный, шифрованный):** темы, паттерны, защиты, эмоции, инсайты, гипотезы, интервенции. Валидируется психологом.

## 5. Голосовой пайплайн

### 5.1 Архитектура обработки голосового сообщения

```
Voice message (Telegram .ogg, до 60 сек)
        │
        ▼
┌──────────────────────┐
│ Preprocessing        │  ffmpeg: нормализация, ресемплинг до 16kHz mono
│                      │  VAD: удаление тишины в начале/конце
└──────────┬───────────┘
           │
    ┌──────┴──────┬─────────────────┐
    ▼             ▼                 ▼
┌────────┐  ┌─────────────┐  ┌──────────────┐
│Whisper │  │ Prosody     │  │ Emotion      │
│large-v3│  │ librosa +   │  │ SpeechBrain  │
│→ text  │  │ parselmouth │  │ or wav2vec2  │
│+ VAD   │  │             │  │              │
└───┬────┘  └──────┬──────┘  └──────┬───────┘
    │              │                 │
    └──────────────┼─────────────────┘
                   ▼
       ┌───────────────────────┐
       │ Voice context bundle  │
       │ + baseline comparison │
       └───────────┬───────────┘
                   ▼
              в Main Agent
```

### 5.2 Извлекаемые фичи

**Просодические (librosa + parselmouth):**
- `duration_sec` — длительность
- `tempo_wpm` — слов в минуту
- `pitch_mean_hz`, `pitch_std_hz` — средняя высота и вариативность
- `energy_mean`, `energy_std` — громкость и динамика
- `jitter`, `shimmer` — микровариации голоса (маркер напряжения/плача)
- `pause_count`, `long_pause_positions[]` — паузы >2 сек и где
- `speech_ratio` — доля времени, когда говорит (vs тишина)

**Эмоциональные (классификатор):**
- `emotion_primary`, `emotion_confidence`
- `arousal` (−1..1) — возбуждение
- `valence` (−1..1) — положительность/отрицательность
- Распределение по базовым эмоциям

**Сравнение с baseline:**
- `deviation_from_baseline` — z-score по каждой фиче относительно нормы клиента

### 5.3 Baseline клиента

**Накопление:**
- Первые 5 голосовых записываются как есть, baseline ещё не считается.
- После 5+ голосовых считается mean/std по каждой фиче → `voice_baselines`.
- Обновляется rolling window (последние 30 голосовых), чтобы ловить изменения.

**Детекция значимых отклонений:**
- z-score > 1.5 по ключевым фичам → флаг «заметное отклонение».
- z-score > 2.5 → флаг «сильное отклонение» + сигнал в Review UI.

### 5.4 Что передаётся в промпт

**Не** «грусть 73%». Структурированные наблюдения:

```
[Voice analysis]
Длительность: 47 сек
Темп речи: медленный (на 30% ниже обычного для этого клиента)
Паузы: 3 паузы >2 сек; длинная (4.5 сек) перед словом «мама»
Голос: тихий, сниженная энергия, признаки дрожания
Эмоциональный профиль (низкая уверенность 0.62):
  - доминирующий: грусть
  - arousal: низкий, valence: отрицательный
Отличие от baseline: заметное снижение энергии,
больше пауз перед темой семьи
```

### 5.5 Правила использования голосовых сигналов

**Уровень 1 — корректировка тона ответа.** В промпте: «учти низкую энергию — не бодри, замедлись, не торопи».

**Уровень 2 — отражение.** Только если согласовано с её методом: «ты сейчас сказала это очень тихо».

**Уровень 3 — сигнал в граф и отчёт.** Тренды, паттерны («замедляется на теме работы»), аномалии.

### 5.6 Safety и голос

**Голосовые маркеры НЕ триггерят кризис автоматически.** Они повышают приоритет текстового анализа. Если текст чистый, а голос тревожный — флаг для психолога, не эскалация.

**Voice cloning risk:** все голосовые хранятся, подозрительные паттерны (резкое изменение акустических характеристик клиента) — алерт.

**Opt-out:** клиент может отключить анализ эмоций, оставив только транскрипцию.

### 5.7 Хранение

```sql

-- voice_features: {
```

## 7. Режимы взаимодействия (структурированные форматы)

Не только открытый чат. Каждый режим — отдельный handler со своим промптом и логикой:

### 7.1 Открытый чат (`open`)
Свободный диалог. Main Agent с полным контекстом. Самый рискованный.

### 7.2 Дневник настроения (`diary`)
Короткая структурированная форма: что чувствую, что произошло, интенсивность 1–10, телесные ощущения. Бот минимально отвечает, сохраняет запись. Низкая нагрузка на LLM.

### 7.3 Проверка домашки (`homework`)
Бот вытягивает активные домашки клиента из графа → конкретные вопросы → фиксирует ответы. Ответы идут в pre-session report.

### 7.4 Подготовка к сессии (`prep`)
За 24 часа до сессии. «Что хочешь обсудить завтра?» → структурированный список → в pre-session report для психолога.

### 7.5 Рефлексия после сессии (`reflection`)
Через 2–3 часа после сессии. «Что унесла?» → инсайты, вопросы, сомнения → потенциальные новые узлы графа.

### 7.6 Режим «только слушай» (`listen_only`)
Клиент нажимает кнопку «мне нужно выговориться». Бот принимает, транскрибирует, сохраняет. Отвечает только в конце минимальным: «услышал, спасибо что поделилась». Никакой интерпретации.

### 7.7 Кнопка «передай психологу»
Не режим, а действие. Отмечает текущий фрагмент как «обсудить на следующей сессии». Попадает в pre-session report.

## 8. GraphRAG: пайплайн запроса

**Вход:** сообщение клиента (текст + опц. voice bundle).

1. **Extract entities** — LLM-проход: persons, emotions, events, markers.
2. **Anchor in graph** — поиск узлов через эмбеддинги + exact match.
3. **Graph walk** — рекурсивный SQL на 2–3 хопа с фильтрами по типам рёбер.
4. **Fetch RAG** — параллельно top-k сегментов лекций.
5. **Build context:**
   ```
   [SYSTEM: voice_document активной версии + red lines]
   [CLIENT MODEL: активные паттерны, гипотезы, свежие инсайты]
   [METHOD: релевантные куски лекций]
   [STYLE: 3–5 её ходов под эту ситуацию]
   [VOICE CONTEXT: если голосовое — структурированные наблюдения]
   [HISTORY: последние N сообщений]
   [MODE: current conversation mode]
   [USER MESSAGE]
   ```
6. **Claude call** (temperature 0.4–0.6).
7. **Post-process.**
8. **Send.**
9. **Async update graph** + baseline голоса + мониторинг привязанности.

### SQL обхода графа

```sql
WITH RECURSIVE graph_walk AS (
  SELECT n.id, n.type, n.name, n.properties, 0 AS depth
  FROM nodes n
  WHERE n.client_id = $1 AND n.id = ANY($2::uuid[])
  
  UNION
  
  SELECT n.id, n.type, n.name, n.properties, gw.depth + 1
  FROM graph_walk gw
  JOIN edges e ON e.from_node = gw.id OR e.to_node = gw.id
  JOIN nodes n ON n.id = CASE WHEN e.from_node = gw.id THEN e.to_node ELSE e.from_node END
  WHERE gw.depth < 3 AND e.type = ANY($3::text[])
)
SELECT DISTINCT * FROM graph_walk ORDER BY depth;
```

## 9. Safety Layer

### 9.1 Детекция кризиса

**Уровень 1:** regex/keyword scan (СУ, селф-харм, насилие, РПП-кризис).
**Уровень 2:** Claude Haiku — подтверждение, категоризация, острота.

**При подтверждении:**
- **Hardcoded** ответ (шаблон согласован с психологом).
- Контакты экстренных служб.
- Уведомление психолога в отдельный Telegram-канал.
- Блок свободного диалога до подтверждения безопасности.
- Запись с флагом `crisis_detected`.

### 9.2 Скоуп-контроль

Классификатор: диагнозы, фармакология, юридические/финансовые вопросы, другие специалисты → вежливый отказ + направление.

### 9.3 Post-processor

**Anti-sycophancy:** детектор «я понимаю тебя», «это нормально», «ты молодец», избыточной валидации, немедленных советов без вопроса.

**Uncertainty check:** если ответ модели содержит высокую долю generic-фраз или не опирается на контекст клиента — регенерация с промптом «скажи что не понимаешь и попроси уточнить, это лучше, чем правдоподобная чушь».

**Scope leak:** ответ не вышел за рамки скоупа.

### 9.4 Мониторинг привязанности (новое)

Еженедельный автоматический анализ метрик клиента:
- рост частоты обращений (>30% неделя к неделе)
- рост средней длины сообщений
- формулировки вроде «ты лучший друг», «не знаю что бы я без тебя делала»
- сокращение упоминаний реальных людей в диалогах с ботом

При триггерах → флаг `attachment_risk` в Review UI для психолога.

### 9.5 Лимиты взаимодействия

- Max 30 минут непрерывного диалога, потом мягкая пауза с напоминанием о границах.
- Max 5 инициированных клиентом сессий в день, дальше — «я буду здесь завтра».
- Периодические (раз в 2 недели) reminder-сообщения: «я инструмент, не заменяю [имя психолога]».

## 10. Supervision UI

**Экраны:**

1. **Dashboard** — общие метрики, свежие флаги, алерты.
2. **Диалоги** — фильтры по клиенту/дате/флагам/режиму. Пометка «good»/«bad»/«revise».
3. **Карточка клиента** — граф (визуализация), таймлайн сессий, активные паттерны, voice-тренды, warning-флаги (привязанность, аномалии голоса).
4. **Pre-session report** — автоматически к каждой живой сессии.
5. **Voice document editor** — версионированный, с diff между версиями, с тестовыми диалогами «до/после» (если ответ бы изменился).
6. **Review queue** — подозрительные ответы, крисисные кейсы, voice anomalies, attachment flags.
7. **Метрики** — кол-во диалогов, распределение режимов, thumbs up/down, флаги.
8. **Weekly system report** — бот пишет сам: «3 раза хотел дать совет, 2 раза не нашёл хода в style-library, вот кейсы — возможно, обновить voice-doc».

Стек: Next.js + Postgres. Авторизация через passkey или магик-линки. Доступ логируется.

## 11. Приватность и безопасность

- TLS везде.
- **Шифрование на уровне приложения** (AES-256-GCM) для PII полей.
- **Отдельная БД** для клиентских данных vs метод/стиль.
- Ротация ключей раз в квартал.
- Логи без PII на уровне warning/error.
- Полное стирание клиента по запросу (включая граф, эмбеддинги, аудио).
- Явное согласие при онбординге (что бот делает, какие данные, кто видит).
- **Никакого файнтюнинга** на клиентских транскриптах.
- Доступ — только психолог и инженер, логируется.
- Шифрованные бэкапы off-site, ограниченный retention.
- Аудио хранится шифрованным, имеет TTL (например, 90 дней).

## 12. Режим «тренажёра»

Перед запуском на реальных клиентах — период, когда бот работает **только с психологом**:

- Она отправляет ему реалистичные сценарии как «клиент».
- Смотрит ответы, правит voice-doc, пополняет style-library.
- Прогоняются кризисные сценарии — проверка, что hardcoded-ответы работают.
- Прогоняются edge-cases: манипуляции, провокации, попытки сменить тему, jailbreak-подобные запросы.

Длительность: 2–4 недели, пока психолог не скажет «готово».

## 14. Критичные «НЕ»

- Не файнтюнить на сырых транскриптах.
- Не запускать без письменного соглашения о супервизии.
- Не обещать технически необеспеченную конфиденциальность.
- Не делать бесконечный чат.
- Не перформировать эмпатию («я переживаю», «я скучал»).
- Не импровизировать в кризисе.
- Не применять авто-извлечённые узлы без валидации.
- Не триггерить кризис только по акустике голоса.
- Не делать эмоциональный классификатор первой версией — сначала базовая просодия.
- Не заменять живые сессии отчётами бота.

### 🔵 MVP-1: Case-study на записях разработчика-клиента (Неделя 7–14)

**Цель этапа:** на единственном доступном клиенте (разработчике проекта) выработать и валидировать онтологию графа. Совместно с психологом: ручная разметка → золотой стандарт → автоматический пайплайн → проверка на golden standard.

**Все работы в этом этапе подчиняются ограничениям раздела 1b** — используются только «закрытые» сессии, разработчик-клиент не пользуется графом для своей терапии, психолог имеет право вето.

#### Этап 1.1 Отбор записей для обработки (Неделя 7)
- [ ] Психолог отмечает записи как `closed_by_therapist=true`

- [ ] Финальный список согласуется на отдельной встрече
- [ ] Обработке подлежат **только** прошедшие фильтр записи

#### Этап 1.2 Транскрипция и подготовка (Неделя 7–8)

- [ ] Сегментация по терапевтическим эпизодам

#### Этап 1.3 Онтология графа — совместная разработка (Неделя 8–9)

- [ ] Первая версия онтологии оформляется как документ
- [ ] Ручная разметка 2–3 эпизодов из разных сессий — психолог + разработчик размечают совместно
- [ ] Корректировка онтологии по результатам разметки

#### Этап 1.4 Золотой стандарт (Неделя 9–10)
- [ ] Совместная ручная разметка всех отобранных сессий

### 🟡 MVP-2: Минимальное сопровождение между сессиями (Неделя 15–22)

**Цель этапа:** Telegram-бот для клиента в **намеренно безопасных режимах** — без открытого терапевтического чата. Базовая функциональность сопровождения: дневник, подготовка/рефлексия к сессиям, передача психологу, напоминания о домашках, вопросы про метод. Первый пилотный пользователь — разработчик-клиент.

#### Этап 4. Базовый бот и safety (Неделя 15–16)

**4.1 Telegram-бот скелет (Неделя 15)**

- [ ] Главное меню с режимами и явными границами

**4.2 Структурированные режимы (Неделя 15–16)**

- [ ] Подготовка к сессии (за 24 часа: «что хочу обсудить»)
- [ ] Рефлексия после сессии (через 2–3 часа: «что унесла»)
- [ ] Только слушай (выговориться, бот не отвечает)
- [ ] Кнопка «передай психологу» на любом сообщении

**4.3 База знаний для клиента (Неделя 16)**
- [ ] Режим «вопрос про метод» — RAG из MVP-0 + voice-doc
- [ ] Жёсткие границы: только информационные ответы, без интерпретаций
- [ ] Если вопрос звучит как личный запрос → перенаправление к психологу

**4.4 Safety Layer v1 (Неделя 16)**
- [ ] Уровень 1: regex/keyword для кризисных маркеров (на русском!)
- [ ] Уровень 2: Claude Haiku классификатор
- [ ] Hardcoded кризисные ответы (согласованы с психологом)
- [ ] Уведомление психолога в отдельный Telegram-канал
- [ ] Локальные ресурсы экстренной помощи: РФ, Самуи/Тайланд
- [ ] Скоуп-контроль: отказ + перенаправление
- [ ] Лимиты взаимодействия (длительность, частота)

#### Этап 5. Pre-session report и напоминания (Неделя 17–18)

**5.1 Автоматический pre-session report (Неделя 17)**
- [ ] Автогенерация за 24 часа до каждой сессии

**7.1 Запуск (Неделя 19)**
- [ ] Разработчик-клиент начинает использовать систему

## 16. Открытые вопросы для обсуждения с психологом

Перед стартом — закрыть эти вопросы:

1. **Срок «закрытости» сессии** — через сколько месяцев после записи сессия попадает в обработку (см. раздел 1b)?
2. **Какие действующие клиенты** могут быть подключены в пилот (не-острые, стабильные, давно в терапии) — короткий список с обсуждением каждого.
3. **Кризисные шаблоны** — конкретные тексты ответов и контакты служб (РФ, Самуи).
4. **Границы скоупа** — список тем, от которых бот отказывается, и формулировка отказа.
5. **Доступные режимы для разных клиентов** — все ли режимы доступны всем, или для каждого клиента психолог настраивает индивидуально.
6. **Отражение голосовых наблюдений** — уместно ли в её методе говорить «ты сейчас сказала это тихо» (в MVP-3).
7. **Частота её супервизии** — ежедневно, раз в 3 дня, раз в неделю?
8. **Где проходит граница** «передать живому психологу» — какие триггеры кроме кризиса?
9. **Имя и характер бота** — у него есть имя? Как он представляется?
10. **Когда бот молчит** — есть ли темы, где он вообще не отвечает, только записывает?
11. **Что делать с voice анализом, если клиент против** — полный opt-out или частичный (в MVP-3)?
12. **Внешний клинический взгляд на двойную роль** — кто будет супервизором конфигурации проекта (раздел 1b)?

## 17. Риски и митигации

| Риск | Митигация |
|------|-----------|
| Суррогатные отношения клиент–бот | Лимиты, reminder-сообщения, мониторинг привязанности |
| Психолог начинает полагаться на бота | Явная договорённость, регулярное обсуждение, метрика «живые сессии не сокращаются» |
| Эффект зеркала (бот усиливает её слепые пятна) | Периодический внешний клинический review (не обязательно технический) |
| Дрейф voice-document без контроля качества | Версионирование, diff, тестовые диалоги «до/после» |
| Voice cloning атака | Логирование всех голосовых, детекция резкого изменения baseline |
| Ложноположительный кризис-триггер | Два уровня детекции, hardcoded ответ всё равно безопасен |
| Ложноотрицательный кризис-триггер | Регулярный аудит логов психологом, метрики false-negative на размеченной выборке |
| Утечка данных клиентов в обучение модели | Только Claude API (не обучается на inference), никакого файнтюна |
| Галлюцинации GraphRAG на разреженном графе | Fallback на обычный RAG + явное «я пока мало о тебе знаю» в первые недели |
| Перегрузка психолога супервизией | Weekly system reports, приоритизация в review queue, метрики «что смотреть в первую очередь» |
| Дрейф моделей Claude между версиями | Pin версии модели, регрессионный прогон golden eval перед обновлением |
| Anthropic content policy блокирует кризисные темы | Тестирование на кризисных диалогах ДО прода, fallback-стратегия |
| Переполнение контекст-окна при долгой работе с клиентом | Memory consolidation: ежемесячные саммари + флаг «активные узлы за 3 мес» |
| SPOF на одном инженере во время инцидента | Runbook аварийного отключения для психолога, документация handoff |
| Психолог уходит из профессии / длительно недоступна | Continuity plan: graceful shutdown с экспортом данных клиентам, режим «психолог в отпуске» |
| Бот закрепляет слепые пятна психолога | Структурированные blind-spot интервью с супервизором, явный список «зон-перенаправлений» |

## 18. Дополнения после ревью v5 (без юридической части)

### 18.1 Расширение Safety Layer: категоризация кризисов (дополняет 9.1)

Кризис — не бинарное событие. Категории, каждая со своим hardcoded-шаблоном и своим каналом уведомления:

- **Active suicidal ideation** (план + средства/срок) → немедленно
- **Passive ideation** («лучше бы меня не было», без плана) → флаг психологу, не эскалация
- **Self-harm / NSSI** — отдельная категория от суицида (разные ресурсы, разный тон)
- **Homicidal / угроза третьим лицам** — отдельный flow
- **Dissociation / panic attack** → grounding-скрипт, не интерпретация
- **Disclosure of abuse** (домашнее насилие, насилие над детьми) — отдельный flow
- **Eating disorder crisis** — специфичные ресурсы (не общая горячая линия)
- **Кризис третьего лица** («моя подруга хочет…») — отдельный путь, отличный от собственного кризиса клиента

### 18.2 Дополнения в раздел 14 «Критичные НЕ»

- Не использовать TTS-голос психолога (усиливает привязанность к боту, а не к психологу).
- Не обновлять voice-document или версию Claude без прогона на golden eval set.
- Не делать единственного инженера SPOF без runbook'а аварийного отключения для психолога.
- Не апгрейдить клиента между MVP-этапами (например, MVP-2 → MVP-3) без явного нового согласия — риски меняются.

### 18.3 Технические оптимизации (внести в Этап 0)

- **Prompt caching (Anthropic)**: voice-doc + style library + последние N сообщений — кандидаты для кэширования. ~90% экономии стоимости, ~80% latency на повторных запросах.
- **Batch API** для оффлайн-обработки лекций и пайплайнов А/Б/В — в 2 раза дешевле.
- **HNSW** вместо ivfflat для pgvector с самого начала — узлы графа быстро растут.
- **Memory consolidation** для долгих клиентов: ежемесячное саммари + флаг «активные узлы за последние 3 месяца», иначе контекст переполнится через 1–2 года.
- **Pin Claude model version + golden eval framework**: 50–100 фиксированных диалогов с эталонными ответами, автоматический прогон при любом изменении voice-doc или модели Claude. Без этого регрессии незаметны.

### 18.4 Multi-therapist ready с самого начала

Во все таблицы метода (`concepts`, `therapist_moves`, `voice_document`, `system_reports`) добавить `therapist_id UUID NOT NULL`. В MVP-0..3 значение зашито на одного психолога, но FK уже есть. Снимает болезненный рефактор, если когда-то появится второй психолог.

### 18.5 Новый режим: `therapist_mode` — бот для самой психолога

Не для клиентов. Психолог использует своего же бота для:
- организации заметок между сессиями;
- «поговорить со своим методом» при подготовке к супервизии;
- поиска по корпусу лекций и собственных размышлений.

Низкий риск, реальная польза для пользователя № 1. Может быть включён уже в MVP-0.

### 18.6 Право клиента на доступ к своему графу

Этический принцип: клиент должен иметь возможность увидеть граф, который система строит о нём. Минимум — read-only вью в боте «что я знаю о тебе», с возможностью пометить узел как «неверно / удалить». Снижает риск отчуждения и улучшает качество графа (клиент исправляет ошибки). Включить в MVP-3.

### 18.7 Onboarding-скрипт первого диалога

Не просто «бот пишет hello». Структурированный onboarding (5–7 экранов), фиксирующий mental model клиента на месяцы вперёд:
1. Что я умею / не умею.
2. Кто видит твои сообщения (психолог — да, всегда).
3. Я НЕ заменяю терапию.
4. Кнопка «передай психологу» — где, как пользоваться.
5. Что в кризисе — звони сюда (локальные контакты).
6. Лимиты использования.
7. Явное «я понял» перед стартом.

Скрипт пишется совместно с психологом, тестируется на разработчике-клиенте.

### 18.8 Скрипты отказов

Для каждого типа отказа (вне скоупа, кризис, лимит, непонятно) — конкретные согласованные с психологом тексты. Не генерируются на лету.

### 18.9 WAI как метрика терапевтического альянса

Working Alliance Inventory (короткая 12-вопросная версия) раз в месяц через бота. Сильнее PHQ-9/GAD-7 как ранний сигнал «терапия страдает», включая искажения от бота. Добавить в MVP-3 рядом со шкалами.

### 18.10 Wearables / passive signals — опциональный канал baseline (MVP-2+)

HRV, сон, активность из Apple Health / Garmin — дешёвый и менее invasive baseline сигнал «как клиент сегодня» по сравнению с просодией. Применим уже в MVP-2, где голоса ещё нет. Только с явного согласия, только агрегаты.

### 18.11 Continuity plan (бизнес-непрерывность)

- **Психолог недоступна / уходит**: процедура graceful shutdown — клиентам экспорт их данных в человекочитаемом виде, бот переходит в read-only.
- **Разработчик недоступен**: runbook для психолога — как отключить бота одной командой/кнопкой.
- **Длительный отпуск психолога**: режим «психолог в отпуске» — бот сообщает клиенту, лимиты сужаются, кризисный канал перенаправляется на дежурного коллегу.

### 18.12 Финансовая модель — определить до MVP-2

Кто платит за инференс/хостинг/Whisper, при каком числе активных клиентов это окупается, есть ли потолок стоимости на клиента. Цель — понять, не создаёт ли модель давление на психолога не отключать «нездоровые» паттерны использования.

### 18.13 Расширение раздела 16 «Открытые вопросы»

- Кто супервизор/коллега для blind-spot review психолога?
- Континьюити-сценарий: что делать, если психолог временно недоступна?
- Культурный контекст клиентов (РФ / Тайланд / другие) — как отражается в voice-doc?
- Доступ клиента к своему графу — насколько детально, в каком виде?
- TTS-голос психолога — явное «нет» или возможны исключения?

### 18.14 Расширение метрик (раздел 13)

- **WAI** раз в месяц (терапевтический альянс)
- **Cost per active client** (Claude API + хостинг)
- **Drop-off по режимам** — какой режим клиенты бросают, какой используют регулярно
- **Time-to-respond психолога** на алерт — целевая SLA должна быть явной (час в MVP-3)
- **Golden eval pass rate** при каждом релизе voice-doc/модели — не должна падать
